## Fixes Applied

### 1. Stale `claimed` host assignments

Classification: functional bug

Problem:

- a plane could reach `applied_generation == generation`
- the latest `apply-node-state` assignment could still remain `claimed`
- dashboard and plane summaries therefore kept reporting false in-flight work

Fix:

- introduced `HostAssignmentReconciliationService`
- reconciler now:
  - marks older `claimed` assignments as `superseded` when a newer assignment already replaced them
  - marks latest `claimed` assignments as `applied` when the plane is already converged
- reconciliation is now called from:
  - `PlaneService` lifecycle transitions
  - `SchedulerService::SchedulerTick()`
  - `PlaneRegistryService`
  - `DashboardService`

## Open Findings

### 1. Controller CLI cannot mutate the live controller DB on `hpc1`

Classification: broken UX/observability

Observed behavior:

- running `comet-controller show-host-assignments` or `scheduler-tick` directly against `/var/lib/comet-node/hostd-state/controller.sqlite` from an SSH shell returns:
  - `sqlite step done failed: attempt to write a readonly database`

Impact:

- operator CLI workflows cannot safely use the live DB in the same way the service process does
- maintenance actions such as manual `scheduler-tick` from shell are unreliable on the live host

Status:

- not fixed in this slice
- needs investigation of DB directory ownership / journaling / service-vs-shell execution path
